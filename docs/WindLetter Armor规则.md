# WindLetter Armor 规则

本文冻结 WindLetter v1 Armor 的文本封装、版本引导字符和 WindBase1024F v1 字表。

## 公共帧

逻辑帧为：

```text
WLA | canonical unsigned LEB128(version) | uint32 length | outer wire JSON UTF-8 | CRC-32
```

- `WLA` 是固定 Magic：`57 4C 41`。
- 版本号是正整数，使用最短、无冗余零组的 unsigned LEB128；v1 编码为 `01`。
- CRC-32 覆盖其前面的完整帧内容。
- 未支持或非规范版本必须在访问私钥前拒绝。

## 标准 Base64 文本

```text
-----BEGIN WIND LETTER-----
<standard padded Base64, 64 ASCII characters per line>
-----END WIND LETTER-----
```

使用 RFC 4648 标准 Base64 和 `=` padding。Base64 本身没有独立版本；解码后读取公共帧中的 `WLA` 和版本。生成器使用 LF，解析器兼容 LF 与 CRLF。

## 風笺文本

```text
-----風笺 起-----
<固定 WLA 引导><64进制版本号>凪<对应版本的 WindBase 正文>
-----風笺 凪-----
```

- 永久冻结的64字符表既用于固定 WLA 引导，也作为版本号的64进制数字表。
- `WLA` 固定编码为 `渢𩗍𩘥𬱶`。
- 字表索引即数字值 `0..63`；版本号采用大端最短表示，版本0无效，禁止前导零。
- `凪`（U+51EA）永久冻结为版本号终止符，不属于版本数字表或 WindBase1024F v1 正文表。
- v1 的版本数字为 `𬱶`，完整引导为 `渢𩗍𩘥𬱶𬱶凪`。
- 解析器读取到 `凪` 后，根据版本选择正文1024字表；未知版本必须在正文解码和访问私钥前拒绝。
- 所有字符计数、截断与换行都按 Unicode 码点处理，不使用 Java `String.length()` 作为符号数。
- 風笺正文按64个 Unicode 码点换行；生成器使用 LF，解析器同时接受 LF 与 CRLF。
- 不对输入执行 Unicode normalization。
